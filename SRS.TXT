# Software Requirements Specification: Vizzo Platform Architecture and Implementation Guidelines

### 1.1 Rendering Topology and Hybrid Display Protocols
The core architectural pattern mandates the deployment of a Single Page Application (SPA) framework integrated with a sophisticated Hybrid Rendering Topology. This design choice is not merely an aesthetic preference but a critical functional requirement designed to balance initial payload delivery speed with search engine discoverability and social media sharing mechanics. The root route of the buyer storefront relies strictly on Client-Side Rendering (CSR) to guarantee maximum responsiveness during catalogue browsing. Once the initial JavaScript payload is parsed and executed by the browser, subsequent navigational transitions within the storefront must execute instantaneously through virtual DOM reconciliation, effectively operating as a zero-latency interface devoid of hard page reloads.
Conversely, the individual product specific routes mandate an exclusive Server-Side Rendering (SSR) architecture. This represents an uncompromising engineering requirement designed to dynamically inject Open Graph tags, specifically utilizing og:title, og:image, and og:description metadata fields. By executing server-side rendering on these highly specific endpoints, the system ensures that when product links are disseminated via messaging platforms, WhatsApp web scraping bots can instantaneously parse the HTML document and generate rich, interactive visual preview cards. This dual-rendering strategy secures optimal rendering performance while preserving the viral sharing loops critical to the platform's organic growth.
### 1.2 State Management and Zero-Friction Data Encapsulation
The state management protocol fundamentally rejects traditional session-based account creation paradigms for end buyers, enforcing a strict Zero-Friction Entry architecture. Cart state continuity and user session persistence rely entirely on the native Local Storage API of the buyer's mobile browser. This decentralized approach to state management permits immediate transactional engagement without the conversion-killing friction of authentication barriers.
To prevent systemic vulnerabilities such as State Desync and deliberate client-side price tampering, the system enforces a stringent Dynamic State Hydration security protocol. It is strictly prohibited architecturally to store volatile product metadata, particularly product names or pricing parameters, within the localized browser storage. The mandated client-side payload schema is highly restricted, consisting solely of unique identifiers and quantitative integers. Upon initialization of the buyer interface, the frontend application engine must asynchronously fetch real-time pricing data via secure API calls from the backend server, which acts as the absolute Source of Truth. The client application then maps this secure server data against the stored local identifiers to compute cart totals dynamically, ensuring total financial accuracy even if a merchant alters a price mid-session.
### 1.3 Network Engineering and Latency Mitigation Protocols
To counteract the severe Latency Drop-off metrics associated with unstable 3G network conditions in the operational region, the platform enforces aggressive optimization algorithms across all visual and data assets. The primary bottleneck identified in user retention is image transmission latency. Therefore, the engineering specification mandates a Client-Side Compression matrix. Before any image asset is transmitted to the server architecture via the merchant application, the frontend must execute a JavaScript-based compression routine, restricting upload payload sizes to a strict maximum of 200 Kilobytes per asset.
On the delivery side, specifically within the buyer storefront, all imagery must be forcibly routed through a server-side compression pipeline that converts files to the highly efficient WebP format encoded at an 80 percent quality ratio. This algorithmic reduction is engineered to shrink the incoming 200 Kilobyte source files down to approximately 40 Kilobytes prior to client delivery. This extreme data compression strategy ensures uninterrupted navigation, fluid scrolling, and instantaneous visual feedback, successfully neutralizing the bounce rate associated with slow internet connectivity.

Positioned immediately beneath the persistent sticky header, the Centralized Search Module deploys a full-width input field driving a highly optimized Real-time Regex Match client-side algorithm. To guarantee immediate sensory feedback and prevent input lag, the system must process user keystrokes locally against the pre-loaded JSON inventory array with a strict, unyielding latency budget that must not exceed 50 milliseconds per execution cycle.
### 4.3 Infinite Grid Generation and Lazy Loading Protocols
The primary product feed utilizes a combination of Horizontal Swipeable Carousels and an underlying Infinite Grid rendering matrix. The internal Sorting Logic Execution relies on a dropdown parameter allowing state modifications such as Newest, Lowest Price, and Highest Price. The UI engineering guidelines completely ban the use of traditional CSS loading spinners or progress bars during these asynchronous fetch events. Instead, the protocol enforces the rendering of Skeleton Screens—pulsing visual wireframes mapping exactly to the dimensions of the unloaded product cards—to significantly compress the psychological metric of Perceived Latency during data hydration cycles.
### 4.4 Product Entity Engineering and Variant Interception Logic
The individual product card UI rendering engine allocates exactly 60 percent of vertical pixel space to the display of the highly optimized, 80 percent WebP-compressed thumbnail image, ensuring maximum visual fidelity against minimal bandwidth expenditure.
The database architecture and subsequent frontend processing explicitly outlaw the construction of Price-Mutating Variants. For example, a single mobile phone entity model with both 128GB and 256GB storage options scaling differently in price cannot be compiled into a single multidimensional array, as this introduces severe complexity into both the merchant dashboard and the buyer's checkout flow. These must be engineered as entirely separate entity entries instantiated via the Rapid Clone Engine. Conversely, purely Cosmetic Flat Variants, such as aesthetic device color, are seamlessly supported via a flat string Tag array attached to the primary item.
When a buyer interacts with the primary "Add to Cart" interaction node for an inventory item possessing these Cosmetic Variants, a strict Interception Protocol overrides the direct Local Storage array insertion. The application instantly pushes a Bottom Sheet Modal into the active DOM view, enforcing a mandatory attribute selection matrix via Radio Chips (e.g., selecting the color "Blue") before the final cart append operation function is programmatically authorized.
### 4.5 SEO Degradation and Tombstone State Protocols
To comprehensively prevent the severe deterioration of Search Engine Optimization (SEO) rankings and the catastrophic routing failure of historical WhatsApp links shared in external peer-to-peer networks, a global Hard Delete Prohibition is enforced across the database architecture. Products removed by the merchant are instead securely migrated to an Is_Archived = True status flag.
If a buyer subsequently navigates to the URL endpoint of an archived or out-of-stock product entity, the system intercepts the standard HTTP 404 response generation. Instead, it executes the Tombstone UI State. The requested product card is rendered visually inert via a global Grayscale CSS filter property overlaid with a critical red status badge explicitly reading [نفد من المخزون]. Immediately beneath this Tombstone UI component, the recommendation engine queries the identical product category tree, instantiating a fresh interactive grid titled exactly [منتجات مشابهة متوفرة حالياً] in an aggressive attempt to recapture and redirect the user's interrupted purchase intent toward available inventory.
### 4.6 Cart State Hydration and Payload Compilation
The traditional multi-stage e-commerce shopping cart pipeline is entirely bypassed within this platform, abandoned in favor of a highly localized payload assembler that terminates rapidly at a single Conversion CTA labeled explicitly [إرسال الطلب].

The interaction constraints of this cart matrix strictly cap unique array elements to a maximum of 15 items per session. Furthermore, the associated custom notes text input node, labeled [الملاحظات], is architecturally truncated at a maximum limit of 150 characters. This specific character constraint is enforced to mathematically prevent the subsequent generated URL string payload from exceeding the maximum byte buffer limitations inherent in legacy mobile browsers during the deep link handover.
### 4.7 Concurrency Risk Mitigation and Zero-State Fallbacks
To mathematically manage concurrency risk in a high-volume distributed system, the architecture injects a silent, rapid API call—the Pre-flight Concurrency Check—milliseconds before processing the final payload dispatch to the WhatsApp protocol. This mechanism polls the server array for the real-time Is_Available boolean status of every single index currently residing in the cart array. If a data conflict is detected (e.g., an item has been toggled out of stock by the merchant while the buyer was actively browsing), script execution instantly halts, and a red error boundary exception forces the user to manually purge the conflicting array item from the DOM before proceeding.
Upon successfully passing the concurrency validation check, the engine triggers the Reverse WhatsApp Payload Generator. This complex function serializes the multidimensional Local Storage array into a highly structured Plaintext string format. It then processes this string via deep URL Encoding algorithms, concatenating the result with the mandatory +249 merchant routing number to construct a seamless, OS-level wa.me Deep Link.
The mandated output template structure generated by this function must conform identically to the following format layout:
مرحباً، أود إتمام هذا الطلب من متجرك:
1x [اسم المنتج 1] - [المتغير إن وجد]
السعر: [السعر اللحظي]
الإجمالي: [المجموع الكلي]
ملاحظات: [النص المدخل]
The serialization script inherently appends the unique store Deep Link URL directly at the termination point of the output string to ensure the receiving merchant possesses absolute, deterministic tracking capability tracing back to the exact data node the buyer initially interacted with.
